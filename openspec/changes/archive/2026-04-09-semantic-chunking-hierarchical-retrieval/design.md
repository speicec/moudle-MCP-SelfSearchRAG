## Context

当前RAG系统采用固定token长度切分策略，存在三大问题：
1. **语义断裂**：在句子中间、表格行内、代码块边界强制切分，破坏语义完整性
2. **检索噪声**：大量低相关性分块污染检索结果，有效信息被淹没
3. **Token浪费**：盲目返回数十个分块，大模型处理无效内容消耗大量token

技术约束：
- TypeScript/Node.js生态
- 需与现有Harness框架集成
- 向量数据库为ChromaDB（可选）或内存存储
- 嵌入模型依赖外部API调用

## Goals / Non-Goals

**Goals:**
- 实现基于语义边界的智能分块，切分点在语义突变处而非固定token数
- 构建父子层级分块结构，小块检索定位，大块返回完整上下文
- 实现Small-to-Big检索策略，精准定位+完整上下文双目标
- 实现相似度断崖检测算法，自动识别语义边界
- 实现分块质量过滤，排除低信息密度分块

**Non-Goals:**
- 不实现自定义嵌入模型（使用现有embedding API）
- 不处理跨文档语义关联（单文档内分块）
- 不实现动态分块大小调整（静态层级策略）
- 不处理加密或特殊格式文档（PDF/文本/图像）

## Decisions

### 1. 语义分块策略

**决定：采用滑动窗口+相似度断崖检测**

方案对比：
- **固定长度切分**：简单但破坏语义边界 ❌
- **句子边界切分**：保留句子完整性，但无法处理段落级语义 ❌
- **滑动窗口+相似度计算**：窗口内计算embedding，相似度断崖处切分 ✅
- **LLM辅助切分**：精度最高但成本高、延迟大 ❌

算法核心：
```
1. 将文档按句子分割
2. 对每个句子生成embedding
3. 滑动窗口(如3句)计算相邻窗口余弦相似度
4. 当相似度 < threshold 时判定为语义断崖，在此切分
5. 合并相邻断崖间句子为语义分块
```

理由：平衡精度与成本，无需LLM调用即可识别语义边界。

### 2. 层级分块结构

**决定：二级层级（小分块→父分块）**

方案对比：
- **单层分块**：无法平衡检索精度与上下文完整性 ❌
- **二级层级**：小块检索，大块返回，简单有效 ✅
- **多级层级（3+级）**：过度复杂，边际收益低 ❌

数据结构：
```typescript
interface HierarchicalChunk {
  id: string;
  content: string;
  embedding: number[];
  level: 'small' | 'parent';
  parentId?: string;  // small chunk指向parent
  childIds?: string[]; // parent包含small chunks
  position: { start: number; end: number };
  qualityScore: number;
}
```

理由：二级层级足够满足Small-to-Big需求，避免过度复杂。

### 3. 相似度断崖检测算法

**决定：阈值检测+梯度验证**

算法步骤：
```
1. 计算相邻embedding余弦相似度序列: [s1, s2, s3, ...]
2. 识别断崖候选: si < threshold (如0.7)
3. 验证梯度: |si - si-1| > gradientThreshold (如0.15)
4. 过滤噪声: 连续N个断崖候选才确认切分点
```

参数配置：
- `similarityThreshold`: 0.7 (低于此值触发断崖检测)
- `gradientThreshold`: 0.15 (相邻相似度变化阈值)
- `minCliffWidth`: 2 (最小断崖宽度，过滤瞬时波动)

理由：阈值+梯度双重验证减少误切分，噪声过滤提高稳定性。

### 4. 分块质量过滤

**决定：多维度质量评分+阈值过滤**

质量指标：
- **信息密度**：分块长度与嵌入向量复杂度的比值
- **重复度**：分块内重复token比例
- **语义完整度**：是否包含完整句子/段落结构
- **关联度**：与文档整体语义的相似度

过滤策略：
- 质量评分 < `qualityThreshold` (如0.3) 的分块标记为低质量
- 低质量分块可选：直接丢弃 或 合并到相邻分块

理由：多维度评分更全面，阈值策略简单可控。

### 5. Small-to-Big检索策略

**决定：两阶段检索+父分块展开**

检索流程：
```
1. 用户查询 → 生成query embedding
2. 在small chunks中检索 → 获取top-k相关小分块
3. 展开父分块 → 获取每个small chunk的parent
4. 合并去重 → 返回完整父分块而非碎片
5. 可选：截断父分块 → 仅返回相关部分
```

优势：
- 小分块检索精度高（语义聚焦）
- 父分块上下文完整（避免碎片化）
- Token消耗可控（返回完整语义单元而非碎片堆）

理由：两阶段检索平衡精度与完整性，避免碎片化输出。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 嵌入计算开销大（每句都需embedding） | 批量嵌入API调用、缓存嵌入结果 |
| 断崖检测参数敏感，可能误切分 | 提供可配置阈值、fallback到固定切分 |
| 父分块过大超出上下文窗口 | 设置父分块最大长度限制、智能截断 |
| 小分块检索命中但父分块不相关 | 父分块质量评分、相关性二次验证 |
| 多模态内容（图像）语义边界难以检测 | 图像作为独立分块，不参与语义切分 |

## Open Questions

1. **断崖检测最优阈值**：需要实验确定最佳similarityThreshold和gradientThreshold值
2. **父分块长度上限**：如何设置避免超token限制？
3. **质量评分权重**：各质量维度权重如何分配？
4. **多模态处理**：图像/表格如何融入层级分块？
